TICKET 13 — Store Events in App State. ***DONE

***Ticket 13 is the most important one conceptually. The setState(prev => [...prev, newEvent]) pattern is something you'll write hundreds of times in React. The reason you use the prev => form instead of just setEvents([...events, newEvent]) is subtle but important — React state updates can be batched, so events inside the handler might be stale by the time it runs. Using the function form guarantees you're always working with the latest version.***

Right now App only logs something.

Now App should store events.

In App.tsx

Add state:

const [events, setEvents] = useState<CareEvent[]>([]);

Then update the handler you passed to LogEventScreen:

function handleLogEvent(event: CareEvent) {
  setEvents((prev) => [...prev, event]);
  console.log("Event logged:", event);
}
Concept learned

Arrays in React state

Also the pattern:

setState(prev => newState)

This prevents stale state bugs.

TICKET 15 — Show Dog Names Instead of IDs  ***DONE

***Ticket 15 is where it gets interesting — you're joining those two arrays together using dogId as the link. This is exactly how relational databases work. The dog?.name syntax with the ? is called optional chaining — it means "if dog exists, give me its name, otherwise give me undefined instead of crashing." You need it because find() can return undefined if no match is found.***

Right now you see:

feed for dog d1

But you want:

Feed for Luna

So you must find the dog that matches the event dogId.

Example:

const dog = dogs.find((dog) => dog.id === event.dogId);

Then render:

<li key={event.id}>
  {event.type} for {dog?.name} at {event.timestamp}
</li>
Concept learned

Relating two pieces of state

You now have:

dogs[]

events[]

linked via

event.dogId

This is basically a tiny relational data model.
